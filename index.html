1. Purpose & Context

This document is meant to:

  • Capture the guiding principles for extension RM assignments and pairing that leadership has been developing
  • Address concrete failure modes from previous extensions (unclear ownership, late/sparse handover, some RMs with 0 fellows, confusion for ~2 weeks after start)
  • Give the Extension Lead a clear, written process so we get:
    • Faster, clearer ownership
    • Better fellow and mentor experience
    • Something that can scale and be iterated on, not re-invented each cycle

2. Problems This Solves

Previous extension cycles experienced several recurring issues:

Ownership Confusion

Some extension RMs spent ~2 weeks unsure of their assignments. Fellows didn't know who to contact. Some RMs ended up with 0 fellows while others were overloaded.

Late/Sparse Handovers

Handoffs happened ad-hoc or not at all. New RMs lacked context on fellow history, mentor dynamics, and ongoing issues. No single source of truth for who owned what.

Supervision Gaps

When RMs changed, SRM oversight sometimes broke. Nobody had full context on a fellow's trajectory across main → extension transitions.

Timing Misalignment

Late hiring and late onboarding meant new RMs were still getting up to speed when they inherited fellows. Handovers happened during crunch periods rather than before.

3. Goals for RM Assignments in Extension

3.1 Continuity Where It Matters

  • Default: If the main-program RM–fellow relationship is strong and the RM has capacity, keep that pairing into extension
  • Make exceptions explicit (documented and communicated) rather than accidental

3.2 Clarity of Ownership from Day One

  • Every extension fellow should know "who is my RM?" and "who is my backup?" before extension starts
  • No RM should start extension unsure of their assignments or with 0 fellows

3.3 Load and Capacity Sanity

  • Use RM-to-fellow ratios that keep RMs below burnout threshold (accounting for main + extension + side projects)
  • Reflect extra cognitive load for cross-time-zone or "megastream" situations (weight as 1.25-1.5× a standard fellow)

3.4 Location and Time-Zone Awareness

  • London-based extension fellows → London RMs where feasible
  • Main-program US fellows remaining in Berkeley → Berkeley RMs, unless there's a very strong continuity reason
  • Allow exceptions (e.g., an RM taking scholars across locations) but treat them as deliberate choices with explicit tradeoffs

3.5 Maintain Supervision Structure

  • SRM → RM → Fellow chain should remain intact
  • When fellows change RMs, the SRM should ideally stay the same (within-pod handoff) or there should be explicit SRM-to-SRM coordination
  • SRMs need context on all fellows in their pod to provide effective oversight

3.6 Evaluation Data is Non-Optional

  • Extension RM assignments must be coupled to standardized fellow & mentor feedback, not just anecdotes
  • This enables: justifying extension RM spend to funders, running probation and promotion decisions fairly
  • Extension RM survey must run on time, with questions aligned to main-program surveys

4. Proposed Structure: SRM Pods as the Handoff Unit

Core Principle: The handoff unit is the SRM pod, not the individual RM.
When fellows transition to extension, they should move within an SRM's supervision cluster whenever possible. Cross-pod handoffs require SRM-to-SRM coordination, not just RM-to-RM.

4.1 Why SRM Pods?

We considered several models for structuring extension handoffs:

Model Pros Cons
Individual RM continuity
(keep same RM regardless of location)		 Maximum relationship continuity Creates cross-timezone burden; breaks location-based SRM supervision; doesn't scale
Paired RMs
(2 RMs share pool of fellows)		 Built-in redundancy; complementary strengths Unclear SRM accountability; adds coordination overhead; "who owns this fellow?" confusion
SRM Pod model
(handoffs stay within SRM's cluster)		 Clean supervision lines; SRM maintains context even if RM changes; scales with org structure May require rebalancing pod sizes for London/Berkeley flow

We recommend the SRM Pod model because it preserves the supervision structure that makes RMs effective. An SRM who has overseen a fellow's main program can maintain oversight into extension even if the day-to-day RM changes—this is better than breaking supervision to preserve an RM relationship.

4.2 How It Works

  • Each SRM oversees a consistent set of RMs across main + extension
  • When a fellow transitions, they move within the pod if possible (same SRM, potentially different RM)
  • Cross-pod handoffs require SRM-to-SRM coordination, not just RM-to-RM—this ensures context transfers at the supervision level
  • London pods should be sized to absorb extension load (given 70% of extensions go to London)

4.3 Paired RMs: When to Use

The paired RM model still has a place, but within the pod structure:

  • Megastreams / multi-mentor setups: Where multiple mentors share a pool and the cognitive load is high, pair a senior/context-heavy RM with another RM to split fellows
  • Onboarding new RMs: Pair new extension RMs with experienced ones for their first cycle
  • Always within same SRM pod: Both RMs report to same SRM, avoiding accountability confusion
  • Document clearly: Who is primary for which fellows; how evaluation and performance input are split
This maintains:
  • Clean supervision lines (SRM → RM → Fellow)
  • Continuity of context at the SRM level even when RMs change
  • Clear accountability for fellow outcomes
  • Sustainability—no single RM is a bottleneck or single point of failure

This maintains:

  • Clean supervision lines (SRM → RM → Fellow)
  • Continuity of context at the SRM level even when RMs change
  • Accountability for fellow outcomes

6. Handoff Timeline

Start 4 weeks before extension begins:

Week -4: Assignment

Extension Lead finalizes RM assignments using flowchart. SRM pods confirmed. All exceptions documented.

Week -3: Session 0 — Async Heads-up

Email/Slack to fellow & mentor: "Here's your extension RM, here's the handoff plan." Both RMs cc'd.

Week -2: Session 1 — New RM Shadows

Existing RM leads the check-in. New RM attends, observes, asks questions, takes notes.

Week -1: Session 2 — New RM Leads

New RM runs the meeting. Existing RM shadows, fills gaps, provides context as needed.

Week 0: Extension Starts — New RM Owns

New RM has full ownership. Old RM available for questions for 2 weeks, then steps back entirely.

7. Pre-Extension Checklist

Extension Lead completes before Day 1:

Fellow Assignments

  • Every extension fellow has a named primary RM
  • Every extension fellow has a named backup/co-RM (or explicit "none needed" note)
  • Every extension fellow has a named supervising SRM
  • Handoff status tracked for each fellow: Not Started In Progress Done
  • All cross-pod handoffs have documented rationale

RM Load Validation

  • Each RM's total load calculated: Main + Extension + Side duties
  • No RM exceeds sustainable threshold (check with SRM)
  • No RM starts extension with 0 fellows (process bug if so)
  • Cross-timezone fellows weighted as 1.25-1.5x load
  • Megastream/multi-mentor fellows weighted as 1.5x load

SRM Pod Readiness

  • Each SRM knows which fellows are entering their pod
  • Each SRM knows which fellows are leaving their pod (and to whom)
  • SRM-to-SRM handoff calls scheduled for cross-pod transfers
  • SRM check-in schedule updated for new pod composition

Systems & Communication

  • Airtable/tracking updated with new RM assignments
  • Extension survey questions reviewed (aligned with main program)
  • Survey launch date set (early enough for probation decisions)
  • Fellow/mentor notification sent (Session 0)

8. Roles & Responsibilities

8.1 Leadership Roles

Research Director (Laura)

  • Explicit remit includes extension outcomes (funding/employment support)
  • Provides global constraints (capacity, ratios, location split, funding expectations)
  • Ensures alignment with Berkeley main-program RMs and survey standards
  • Co-designs principles; delegates operationalization to Extension Lead

London Leadership (Matthew)

  • Historical context on what went wrong (late hiring, handover chaos, no SSOT)
  • Big picture targets for hiring and capacity
  • Strategy + design focus; less day-to-day extension ops
  • Preferences for onboarding timing (e.g., November for winter extensions)

8.2 Operational Roles

Extension Lead (Henning)

  • Owns assignment scheme end-to-end for London (and most extension RMs globally)
  • Runs the flowchart for each fellow
  • Ensures checklist complete before Day 1
  • Tracks red flags during extension
  • Delegates execution (Patrick for extension/founder support; other RMs for survey ops, Airtable, handover audits)

SRMs

  • Validate RM loads within their pod
  • Participate in cross-pod handoff calls
  • Maintain oversight continuity during transitions
  • Flag capacity/sustainability concerns
  • Ensure handoff quality in their pod

Outgoing RM

  • Prepare handoff notes (fellow context, mentor dynamics)
  • Lead Session 1, shadow Session 2
  • Available for questions for 2 weeks post-handoff
  • Flag any concerns about fellow trajectory

Incoming RM

  • Review handoff notes before Session 1
  • Shadow Session 1, lead Session 2
  • Confirm ownership at Session 3+
  • Raise concerns early, not after problems compound

9. Red Flags to Track

Signals that should trigger mid-extension review:

Process Bugs (fix the system):
  • RM starts extension with 0 fellows
  • Fellow doesn't know who their RM is 1+ week into extension
  • RMs confused about paired-stream ownership
  • Handoff sessions didn't happen (went straight to "new RM owns")
Capacity Issues (may need reassignment):
  • Recurrent overtime/burnout reports from specific RMs
  • Cross-timezone load creating consistent late nights
  • RM struggling with extension + main + side duties combined
Healthy Signals:
  • Fellows report smooth transition in survey
  • SRMs have context on all fellows in their pod
  • No "who owns this fellow?" confusion
  • Handoff rationale documented and retrievable

10. Location-Specific Considerations

Scenario Default Exception Criteria
Berkeley main → Berkeley extension Keep RM if capacity allows Performance issues, RM preference, load balancing
Berkeley main → London extension Handoff to London RM Very strong relationship + fellow preference + RM capacity for cross-TZ
London main → London extension Keep RM if capacity allows Same as Berkeley→Berkeley
London main → Berkeley extension Handoff to Berkeley RM Rare; document rationale clearly

Note: With 70% of extensions in London, London SRM pods should be sized to absorb this load. Berkeley pods "export" fellows but maintain capacity for the 30% staying.

Quick Reference: The 3 Questions

Before extension starts, every fellow should be able to answer:

  1. Who is my RM?
  2. Who is my backup/co-RM? (if applicable)
  3. Who is my SRM? (for escalation)

If any fellow can't answer these on Day 1, that's a process bug to fix immediately.
