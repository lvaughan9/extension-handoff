🔄 Extension Handoff Process

MATS Research Management — v1.0

Core Principle: SRM Pods Stay Intact

The handoff unit is the SRM pod, not the individual RM.
When fellows transition to extension, they should move within an SRM's supervision cluster whenever possible. Cross-pod handoffs require SRM-to-SRM coordination, not just RM-to-RM.

This maintains:

  • Clean supervision lines (SRM → RM → Fellow)
  • Continuity of context at the SRM level even when RMs change
  • Accountability for fellow outcomes

Handoff Timeline

Start 4 weeks before extension begins:

Week -4: Assignment

Extension Lead finalizes RM assignments using flowchart. SRM pods confirmed. All exceptions documented.

Week -3: Session 0 — Async Heads-up

Email/Slack to fellow & mentor: "Here's your extension RM, here's the handoff plan." Both RMs cc'd.

Week -2: Session 1 — New RM Shadows

Existing RM leads the check-in. New RM attends, observes, asks questions, takes notes.

Week -1: Session 2 — New RM Leads

New RM runs the meeting. Existing RM shadows, fills gaps, provides context as needed.

Week 0: Extension Starts — New RM Owns

New RM has full ownership. Old RM available for questions for 2 weeks, then steps back entirely.

Pre-Extension Checklist

Extension Lead completes before Day 1:

Fellow Assignments

  • Every extension fellow has a named primary RM
  • Every extension fellow has a named backup/co-RM (or explicit "none needed" note)
  • Every extension fellow has a named supervising SRM
  • Handoff status tracked for each fellow: Not Started In Progress Done
  • All cross-pod handoffs have documented rationale

RM Load Validation

  • Each RM's total load calculated: Main + Extension + Side duties
  • No RM exceeds sustainable threshold (check with SRM)
  • No RM starts extension with 0 fellows (process bug if so)
  • Cross-timezone fellows weighted as 1.25-1.5x load
  • Megastream/multi-mentor fellows weighted as 1.5x load

SRM Pod Readiness

  • Each SRM knows which fellows are entering their pod
  • Each SRM knows which fellows are leaving their pod (and to whom)
  • SRM-to-SRM handoff calls scheduled for cross-pod transfers
  • SRM check-in schedule updated for new pod composition

Systems & Communication

  • Airtable/tracking updated with new RM assignments
  • Extension survey questions reviewed (aligned with main program)
  • Survey launch date set (early enough for probation decisions)
  • Fellow/mentor notification sent (Session 0)

Roles & Responsibilities

Extension Lead (Henning)

  • Owns assignment scheme end-to-end
  • Runs the flowchart for each fellow
  • Ensures checklist complete before Day 1
  • Tracks red flags during extension
  • Delegates execution (survey ops, Airtable, etc.)

SRMs

  • Validate RM loads within their pod
  • Participate in cross-pod handoff calls
  • Maintain oversight continuity during transitions
  • Flag capacity/sustainability concerns
  • Ensure handoff quality in their pod

Outgoing RM

  • Prepare handoff notes (fellow context, mentor dynamics)
  • Lead Session 1, shadow Session 2
  • Available for questions for 2 weeks post-handoff
  • Flag any concerns about fellow trajectory

Incoming RM

  • Review handoff notes before Session 1
  • Shadow Session 1, lead Session 2
  • Confirm ownership at Session 3+
  • Raise concerns early, not after problems compound

Red Flags to Track

Signals that should trigger mid-extension review:

Process Bugs (fix the system):
  • RM starts extension with 0 fellows
  • Fellow doesn't know who their RM is 1+ week into extension
  • RMs confused about paired-stream ownership
  • Handoff sessions didn't happen (went straight to "new RM owns")
Capacity Issues (may need reassignment):
  • Recurrent overtime/burnout reports from specific RMs
  • Cross-timezone load creating consistent late nights
  • RM struggling with extension + main + side duties combined
Healthy Signals:
  • Fellows report smooth transition in survey
  • SRMs have context on all fellows in their pod
  • No "who owns this fellow?" confusion
  • Handoff rationale documented and retrievable

Location-Specific Considerations

Scenario Default Exception Criteria
Berkeley main → Berkeley extension Keep RM if capacity allows Performance issues, RM preference, load balancing
Berkeley main → London extension Handoff to London RM Very strong relationship + fellow preference + RM capacity for cross-TZ
London main → London extension Keep RM if capacity allows Same as Berkeley→Berkeley
London main → Berkeley extension Handoff to Berkeley RM Rare; document rationale clearly

Note: With 70% of extensions in London, London SRM pods should be sized to absorb this load. Berkeley pods "export" fellows but maintain capacity for the 30% staying.

Quick Reference: The 3 Questions

Before extension starts, every fellow should be able to answer:

  1. Who is my RM?
  2. Who is my backup/co-RM? (if applicable)
  3. Who is my SRM? (for escalation)

If any fellow can't answer these on Day 1, that's a process bug to fix immediately.
